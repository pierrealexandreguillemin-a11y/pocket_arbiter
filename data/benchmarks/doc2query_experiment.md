# Doc2Query Experiment — DISABLED

## Setup

- 2232 synthetic questions (2 per chunk, 1116 chunks)
- Generated by Gemma 3 4B IT (NF4) on Google AI (questions_v5.jsonl)
- Embedded with EmbeddingGemma-300M using format_query() (query prompt)
- Stored in synthetic_queries table, searchable via cosine
- 5th RRF channel with configurable weight, returns child_ids (max score per child)
- Standard: Doc2Query (Nogueira 2019), InPars (2022)

## Root Cause

Le canal synthetic trouve des chunks thematiquement proches mais sur les
**mauvaises pages**. Les "new discoveries" (70%) sont des faux positifs
qui prennent la place des vrais resultats dans le top-k via le RRF.

Contrairement aux narrative rows (+12.3pp tab), les synthetic queries ne
ciblent pas les bonnes pages parce que :

1. Les questions ont ete generees pour le **SFT** (fine-tuning generation),
   pas pour le retrieval. Le SFT cherche l'exhaustivite thematique, le
   retrieval cherche la discrimination semantique.

2. Le GS mesure le recall par **page** — un chunk thematiquement correct
   mais sur la mauvaise page compte comme un miss. Les synthetic queries
   attirent des chunks du meme theme (ex: "forfait" chapitre 3) au lieu
   du chunk exact (ex: "forfait" page 47 article 6.2).

3. EmbeddingGemma-300M a deja une bonne couverture semantique sur le
   cosine standard (59.4% baseline). Le "fossé semantique" entre le
   langage administratif et le langage naturel n'est pas le bottleneck.

## Decision

Canal 5 **DISABLED** (w=0.0). Les 2232 embeddings restent dans la DB
(table synthetic_queries) pour des experimentations futures.

Le probleme n'est pas la qualite des questions (audit OK, 87% independance
lexicale) ni la complementarite du canal (70% unique). C'est que les
"discoveries" pointent vers les mauvais chunks.

## Pistes futures (si re-explore)

1. **Filtrage par page** : ne retourner que les synthetic queries dont la
   page matche une page deja trouvee par cosine/BM25 (confirmation, pas
   decouverte).
2. **Regenerer pour retrieval** : prompt specifique "genere une question
   dont la reponse est UNIQUEMENT dans ce passage" au lieu du prompt SFT.
3. **Score threshold** : ne garder que les synthetic matches avec cosine
   > 0.7 (haute confiance) au lieu de tout passer au RRF.
